# Docker 區網部署設計（Windows 主機、單一入口反向代理）

**日期：** 2026-04-09  
**狀態：** 已與需求方確認（腦力激盪 §1–§4）  
**範圍：** 在安裝 Docker 的 Windows 電腦上以容器執行 Life-Course REMBG（Vite/React 前端 + FastAPI/rembg 後端），並讓**同一區網**內其他裝置透過**單一 HTTP 埠**存取。不包含對網際網路公開服務，也不包含 HTTPS（可另案追加）。

---

## 1. 背景與目標

- **使用者情境：** 開發者擁有一台已安裝 Docker 的 Windows 電腦，希望將本專案以容器方式跑在該電腦上。
- **連線範圍：** 除本機外，**區網內其他裝置**（手機、其他電腦）也需能使用完整功能。
- **非目標：** 取代或複製 Zeabur 雙服務部署流程（現有 Zeabur 文件可並存）；不在本設計內導入 uv／Poetry（後端依賴以 `pip` + `requirements.txt` 為準）。

**成功畫像：** 區網內任一台裝置的瀏覽器開啟 `http://<Windows區網IPv4>:<對外埠>/`，可完成與本機開發時相同的核心流程（預覽、去背、下載 PNG），且可透過同源路徑確認後端健康狀態。

---

## 2. 整體架構（§1）

系統由三個**邏輯角色**組成；實作上收斂成 **兩個長駐容器服務**（見 §3）。

| 角色 | 職責 |
|------|------|
| **反向代理（對外唯一入口）** | 對主機／區網僅暴露**一個 HTTP 埠**（預設語意為 **80**；若被占用則改例如 **8080**）。提供前端靜態資源，並將 API 流量轉送至後端。 |
| **後端** | FastAPI + rembg + uvicorn，僅在 Docker **內部網路**監聽（例如 `:8000`），**不**對區網額外開埠。 |
| **前端靜態檔** | 以 `npm run build` 產物提供；**不**對區網暴露 Vite dev server。 |

**同源 API：** 瀏覽器只與「目前網頁同一主機、同一埠」通訊；API 請求走 **`/api/...`**（以及必要時 **`/health`** 等路徑），由反向代理轉發。因此**不需**在建置時寫入區網 IP，並可避免因雙網域帶來的 CORS 設定負擔（與 Zeabur 前後端分離網址的情境不同）。

**Windows／Docker：** 容器內為 Linux 映像屬預期做法。區網可連的前提包含：Docker **將對外埠發佈到主機的 `0.0.0.0`**（或等同語意），且 Windows **防火牆**允許該埠於**私人網路**連入（細節見 §5）。

## 3. 容器與建置流程（§2）

### 3.1 Compose 服務

| 服務名稱（建議） | 說明 | 對區網暴露埠 |
|------------------|------|----------------|
| **`backend`** | 映像內安裝 `backend/requirements.txt`，啟動命令為 uvicorn 監聽 `0.0.0.0:8000`（埠號可固定為實作約定，但僅內網可達）。 | 否 |
| **`web`** | **多階段映像**：Stage A 使用 Node 執行 `npm ci` 與 `npm run build`；Stage B 使用 Nginx，拷貝 `dist` 與站台設定。對外對應主機上的單一 HTTP 埠。 | 是 |

**依賴安裝：** 後端以 **pip** 為準；Python 版本建議與專案 README 對齊（例如 3.11），於實作計畫中寫死於 Dockerfile。

### 3.2 前端建置時的 API 基底 URL

在「反向代理 + 同源」部署下，目標為：執行期請求指向**目前頁面的同一來源**下的 `/api/...`。實作時需對照 `frontend` 內既有的 `VITE_API_BASE_URL` 使用方式：若程式在變數為空時改走相對路徑，則建置時可不設定或設為空；否則在映像建置階段採用與同源策略一致的明確設定。**具體鍵值在實作計畫階段依程式碼定稿**，本設計不預先假設除「不得寫死區網 IP」以外的額外約束。

### 3.3 Nginx 與後端路由對齊

`proxy_pass`、路徑是否需 **strip prefix** 或保留 **`/api`**，必須與下列兩者一致後再定稿：

1. FastAPI 應用實際掛載的路由前綴（例如是否含 `/api`）。
2. 本機開發時 Vite 對 `/api` 的代理規則（路徑如何轉到 `localhost:8000`）。

本設計只規定**對使用者可觀察到的行為**：瀏覽器僅對單一來源發出請求，且去背與健康檢查在區網與本機均可運作。

### 3.4 啟動順序（可選強化）

Compose 中 `web` 可宣告 `depends_on: backend`。可選：為 `backend` 設定 **healthcheck**，降低啟動初期 Nginx 已就緒但後端尚未監聽而短暫回 **502** 的機率。

## 4. 錯誤處理與觀測（§4）

| 情境 | 預期 |
|------|------|
| 後端未就緒或異常 | 經代理可能回 **502／504**；屬可接受故障表現。自訂 Nginx 錯誤頁為**可選**。 |
| 上傳超過大小限制 | 維持專案現有 **10 MB** 限制與後端行為。 |
| rembg／推理失敗 | 維持後端既有錯誤回應；本設計不新增佇列、重試背景任務。 |

**日誌：** 以 `docker compose logs`（或專案統一文件中的指令）檢視 `web` 與 `backend`。

**健康檢查：** 沿用 **`GET /health`**。實作時應保證在「使用者實際使用的同一主機與埠」下可透過 Nginx 轉發取得（例如對應到後端 `/health`），以利手機或 `curl` 驗證。

## 5. 區網連線與 Windows 防火牆（§3）

- **連線條件：** 用戶端與 Windows 主機須在同一區網（同一 Wi‑Fi／網段）。
- **位址格式：** `http://<區網IPv4>:<對外埠>`。區網 IP 可透過 Windows `ipconfig` 查詢；DHCP 可能變更位址，故障排除時應優先確認 IP 是否仍有效。
- **埠發佈：** Compose 的 `ports` 映射不得僅綁 `127.0.0.1`，否則區網無法連入；應使用對區網有意義的綁定（實作計畫中給出建議寫法）。
- **防火牆：** 為實際對外的 TCP 埠新增「私人網路」允許規則；不建議在標記為公用網路的設定下大開連入。

**範圍外：** 對網際網路公開、HTTPS、憑證管理、路由器埠轉發，均不在本設計內；若未來需要，應另立安全與維運需求。

---

## 6. 驗收標準

1. **本機瀏覽器：** `http://localhost:<對外埠>/` 可完成上傳、去背、下載 PNG。
2. **區網第二裝置：** `http://<區網IP>:<對外埠>/` 完成相同流程。
3. **健康檢查：** 經同一主機與埠（Nginx 轉發）存取 `/health`（或與實作對齊的等同路徑）得到預期就緒回應。
